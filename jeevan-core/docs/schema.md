# Jeevan ŌĆö Database Schema

PostgreSQL 16, evolved exclusively through **Flyway** migrations in
`jeevan-core/src/main/resources/db/migration`. No `ddl-auto` generation ŌĆö
Hibernate runs in `validate` mode only.

Conventions:
- All instant columns are `timestamptz` stored in **UTC**.
- `doctor_availability.start_time` / `end_time` are bare `TIME` values ŌĆö **local
  clinic time**, interpreted in the clinic timezone (`Asia/Kolkata` by default).
- Enum-like columns are `VARCHAR + CHECK` rather than native PG enums, so they map
  cleanly to JPA `@Enumerated(STRING)` and pass Hibernate `validate`.
- Identity columns use `GENERATED BY DEFAULT AS IDENTITY` (lets seed data supply
  explicit ids while runtime inserts auto-generate).

## Tables

### `users`
Patient accounts (login identity = `email`, unique). `role` Ōłł {PATIENT, DOCTOR,
ADMIN} ŌĆö only PATIENT is used by real accounts. `email_verified` defaults `TRUE`
(log-only path); the app sets it `FALSE` at registration when email verification
is enabled.

### `email_verification_tokens`
Opaque single-use tokens for the real-email path. A user may accumulate rows over
time (resends); only the newest unconsumed, unexpired token is valid.
`consumed_at` is null until used.

### `doctors`
Seed data only ŌĆö no doctor login, no admin API. Indexed on `specialty` for the
filter dropdown.

### `doctor_availability`
Recurring weekly windows. **Multiple rows per doctor per weekday are allowed**, so
a mid-day break is two windows on the same day (e.g. 10:00ŌĆō13:00 and 14:00ŌĆō17:00).
`CHECK (end_time > start_time)`. There is no one-off/holiday table ŌĆö recurring only.

### `appointments`
The booked slot. `appointment_status` Ōłł {CONFIRMED, CANCELLED},
`notification_status` Ōłł {PENDING, SENT, FAILED}. Times are UTC `timestamptz`.

### `appointment_history`
Append-only audit log; one row per state change (creation, cancellation,
notification-status updates). `old_status` is null on creation.

## The duplicate-booking guarantee (most important)

```sql
CREATE UNIQUE INDEX uq_appt_doctor_slot_active
    ON appointments (doctor_id, start_time)
    WHERE appointment_status <> 'CANCELLED';
```

This **partial** unique index is the authoritative, race-proof guarantee that a
doctor cannot have two *live* bookings at the same slot start. Because the
predicate excludes `CANCELLED` rows, a cancelled appointment stops participating
in the constraint, so the freed slot can be rebooked. At the application layer
(step 5) a pessimistic lock serialises concurrent attempts on the same slot, and a
constraint violation is translated into a clean `409 SLOT_ALREADY_BOOKED` ŌĆö never
a raw stack trace.

There is **no `slots` table.** Available slots are generated dynamically at request
time from `doctor_availability` minus non-cancelled `appointments` (30-minute
fixed slots, 30-day booking window).
