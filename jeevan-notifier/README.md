# jeevan-notifier

Python FastAPI worker handling **asynchronous side-effects only** ‚Äî notifications. It
reacts to bookings that have **already been committed** by the core; it owns no
appointment data and never decides whether a booking is valid.

- **Python 3.12, FastAPI**; **pika** (blocking consumer on a background thread).
- Consumes `appointment.booked`, `appointment.cancelled`, `verification.requested`
  from `notifier.appointment-events`; publishes `notification.sent` back.
- **Pluggable delivery:** logs by default; sends via MailHog/SMTP when enabled.

## Run

Easiest is the root `docker compose up`. Standalone:

```bash
docker build -t jeevan-notifier .
docker run --rm --network <compose-network> --env-file ../.env -p 8000:8000 jeevan-notifier
```

The consumer reconnects if RabbitMQ isn't up yet.

## HTTP surface

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/health` | Liveness |
| GET | `/notifications` | In-memory log of processed events (proof of work) |
| GET | `/docs` | Swagger UI (auto-generated by FastAPI) |

## Behavior

- On an appointment event: render the notification ‚Üí record it in the in-memory store
  ‚Üí publish `notification.sent` with `status = SENT` (or `FAILED` on error).
- On a verification event: email the verification link (no `notification.sent` ‚Äî there
  is no appointment to attach it to).
- Errors are caught per-message: a `FAILED` result is published and the message is
  dropped (no dead-letter / retry ‚Äî out of scope).

## Config (env)

| Variable | Default | Effect |
|----------|---------|--------|
| `NOTIFIER_EMAIL_ENABLED` | `false` | `true` ‚Üí send via MailHog SMTP; else log only |
| `MAIL_HOST` / `MAIL_PORT` / `MAIL_FROM` | `mailhog` / `1025` / ‚Ä¶ | SMTP target |
| `CLINIC_TIMEZONE` | `Asia/Kolkata` | Timezone used to render human-friendly times in emails |
| `RABBITMQ_*`, `RABBITMQ_EXCHANGE`, `RK_*`, `RABBITMQ_NOTIFIER_QUEUE` | see `.env.example` | Broker topology (must match the core) |
